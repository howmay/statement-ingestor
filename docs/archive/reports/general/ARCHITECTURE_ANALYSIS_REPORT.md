# gmail-expense-parser 專案架構分析報告

## 📊 專案概覽

**專案狀態**: 功能完整，測試覆蓋率 79%，44 個測試全部通過
**主要功能**: 從 Gmail 提取 PDF 收據，使用 LLM 解析，輸出 CSV 格式
**當前版本**: 0.1.0 (Alpha)

## 🏗️ 架構分析

### 模塊結構
```
src/
├── auth/           # Gmail API 認證 (191 行，77% 覆蓋率)
├── bank_parsers/   # 銀行特定解析器 (409 行，平均 90%+ 覆蓋率)
├── fetch/          # 郵件和附件下載 (335 行，平均 81% 覆蓋率)
├── llm/            # LLM 解析和文本處理 (452 行，平均 90% 覆蓋率)
├── ocr/            # OCR 處理 (214 行，56% 覆蓋率)
├── output/         # CSV 輸出 (152 行，90% 覆蓋率)
├── pdf/            # PDF 文本提取 (225 行，63% 覆蓋率)
├── utils/          # 工具函數 (799 行，平均 77% 覆蓋率)
├── app.py          # 主應用程序 (322 行，76% 覆蓋率)
└── config.py       # 配置管理 (41 行，66% 覆蓋率)
```

### 依賴關係分析

#### 核心依賴鏈
```
main.py → app.py → [所有模塊]
```

#### 模塊間依賴
1. **app.py** 依賴所有其他模塊
2. **bank_parsers** 獨立，僅被 llm/parse_receipt.py 調用
3. **utils** 被多個模塊使用，特別是 retry 和 cache
4. **ocr** 模塊使用率低，僅在特定銀行解析時使用

### 架構優點
1. **模塊化設計**: 清晰的職責分離
2. **插件式架構**: bank_parsers 使用工廠模式，易於擴展
3. **錯誤處理**: 完善的 retry 機制和錯誤處理
4. **緩存機制**: 支持 PDF 提取和 LLM 結果緩存
5. **並行處理**: 支持多線程處理 PDF 和 LLM 請求

### 架構問題

#### 1. 循環依賴風險
- `app.py` 導入所有模塊，但某些模塊之間可能存在隱性依賴
- `utils` 模塊被廣泛使用，但某些功能過於通用

#### 2. 性能瓶頸
- **PDF 處理**: 每個 PDF 2-3 秒，使用多層 fallback 機制
- **LLM 調用**: API 延遲和 token 限制
- **網絡請求**: Gmail API 和 OpenAI API 的網絡延遲

#### 3. 測試覆蓋率不均
- 核心模塊覆蓋率高 (80-100%)
- 邊緣情況和錯誤處理覆蓋率低
- OCR 模塊覆蓋率僅 56%

#### 4. 配置管理
- 環境變量和配置文件混合使用
- 缺乏配置驗證和類型檢查
- 敏感信息 (API keys) 處理方式需改進

## ⚡ 性能分析

### PDF 處理性能 (主要瓶頸)
```
提取策略 (優先級):
1. pypdfium2 (最快) - C++ 庫，但需要安裝
2. pdftotext (poppler-utils) - CLI 工具，非常快
3. pdfplumber (Python) - 準確度高，速度中等
4. PyPDF2 (fallback) - 最慢，兼容性最好
```

**問題**: 每次嘗試失敗後才嘗試下一種方法，浪費時間
**建議**: 根據文件特徵選擇最合適的提取方法

### LLM 處理性能
```
優點:
- 支持並行處理 (ThreadPoolExecutor)
- 結果緩存機制
- 請求重試和錯誤處理

缺點:
- 每次調用都需要完整的上下文
- 沒有請求批處理
- 缺乏本地模型優先級
```

### 網絡請求性能
```
Gmail API:
- 郵件搜索和下載串行處理
- 缺乏連接池重用
- 沒有請求合併

OpenAI API:
- 每個請求獨立
- 沒有請求隊列管理
- 缺乏速率限制處理
```
