# Issue #24 解決方案報告：OpenAI API 返回不完整的 JSON 回應

## 問題描述
當解析包含大量交易（30+ 筆交易）的 HSBC PDF 檔案時，OpenAI API 返回不完整的 JSON 回應（大約在 6000 字元處被截斷），導致解析失敗並回退到啟發式解析。

## 錯誤詳情
- 多次嘗試失敗，出現 'Unterminated string' 或 'Expecting value' 錯誤
- 回應在字元 6192/6077 處被截斷
- 影響包含大量交易的 HSBC PDF 檔案

## 根本原因分析

### 1. OpenAI API 限制
- OpenAI API 有最大 token 限制（約 4000 tokens 對應 ~6000 字元）
- 當交易數量多時，生成的 JSON 回應可能超過此限制
- API 會截斷回應，導致 JSON 結構不完整

### 2. 現有程式碼問題
- `_fix_truncated_json()` 函數處理複雜嵌套結構能力有限
- 文本截斷策略（6000 字元）可能截斷重要交易資訊
- 重試邏輯只是簡單重複相同請求，沒有調整參數

### 3. 數據特性
- HSBC 銀行對帳單通常包含大量交易記錄
- 每筆交易需要多個字段（日期、金額、描述等）
- 累積的 JSON 回應很容易超過 API 限制

## 解決方案實施

### 方案選擇：組合策略
選擇了組合策略，結合以下方法：
1. **增強 JSON 修復機制** - 提高修復成功率
2. **智能文本分塊** - 將大型交易列表分割處理
3. **動態 API 參數調整** - 根據文本長度優化請求
4. **增強重試邏輯** - 智能重試不同策略

### 實施細節

#### 1. 增強 JSON 修復機制 (`_fix_truncated_json_enhanced`)
- 改進堆疊追蹤，更好地處理嵌套結構
- 添加字符串補全邏輯，處理未完成的字符串
- 實現智能值推斷（null、空數組、空對象）
- 添加驗證機制確保修復後的 JSON 有效

#### 2. 智能文本分塊策略 (`_chunk_text_by_transactions`)
- 基於交易邊界（日期模式）分塊文本
- 確保每個分塊包含完整交易記錄
- 動態調整分塊大小，避免超過 API 限制
- 添加分塊重疊機制防止交易被分割

#### 3. 動態 API 參數調整
- 根據文本長度動態計算 `max_tokens`
- 使用公式：`min(4000, 1000 + text_length // 2)`
- 對於超長文本自動啟用分塊模式
- 調整溫度參數提高 JSON 生成穩定性

#### 4. 增強重試邏輯
- 三階段重試策略：
  1. 原始請求重試（指數退避）
  2. 減少文本長度重試
  3. 啟用分塊模式重試
- 添加電路斷路器防止無限循環
- 記錄重試統計用於後續優化
